# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## What this repo is

Companion / benchmarking scaffolding for the **braincoder** paper (de Hollander et al., *Braincoder: Easy and rapid modeling of neural representations, forwards and backwards*; draft lives at `paper/Braincoder paper.docx`). It is *not* a pip-installable package — there is no `setup.py`, no `environment.yml`, and no top-level `__init__.py`. The repo holds scripts, configs, BIDS-app wrappers and notebooks that produce the paper's **"Comparison to existing visuospatial PRF packages"** section by:

1. Synthesizing PRF (population receptive field) datasets via the `garikoitz/synthprf` container, BIDS-app style.
2. Fitting those datasets with several competing PRF packages: **braincoder**, **aPRF / prfpy**, **mrVista**, **popeye**, **AFNI**.
3. Comparing estimated parameters against ground truth and against each other, and benchmarking runtime (CPU vs GPU vs A100) — the speed/accuracy figures in the paper.

This whole architecture follows the **Lerma-Usabiaga et al. 2020** PRF-validation framework (PLoS Comp Biol; the paper cites it explicitly). That framework specifies the BIDS-app interface implemented here: each fitter is a `prfanalyze-{pkg}` container that consumes a synthesized BIDS dataset and writes per-parameter NIfTIs into `derivatives/`. Braincoder is the one fitter in this lineup that comes from the authors themselves — the repo wraps it into that same interface (`prfsynth/fit/braincoder/docker_package/`) so the comparison is apples-to-apples.

The braincoder runtime depends on the user's own `braincoder` package (`~/git/braincoder`) installed in a TF2 conda env — the paper repo just wires data in/out and chooses configs.

## Data layout (BIDS, "prfsynth" convention)

All scripts assume the BIDS root at:

- Local:   `/data/ds-prfsynth/BIDS`
- Cluster: `/shares/zne.uzh/gdehol/ds-prfsynth/BIDS`

The dataset name (also the BIDS participant label, also the prfsynth config identifier) is one of:

| Identifier         | Meaning                                                              |
|--------------------|----------------------------------------------------------------------|
| `verysmall`, `small`, `smallgrid`, `mediumgrid`, `largegrid` | Synthetic grids, generated by prfsynth |
| `vanes2019`        | Real fMRI from Van Es et al. 2019 (single voxel timeseries → fake NIfTI) |
| `vanes2019split`   | Split-half version of vanes2019                                      |

The synthesizer writes per-subject `sub-{id}/ses-1/func/sub-{id}_ses-1_task-prf_acq-normal_run-01_bold.nii.gz` plus stimulus apertures under `BIDS/stimuli/`. Every fitting package writes its outputs to `BIDS/derivatives/prfanalyze-{package}[.{variant}]/sub-{id}/ses-1/`. Output files are per-parameter NIfTIs: `sub-{id}_ses-1_task-prf_{par}.nii.gz` where `par ∈ {centerx0, centery0, sigmamajor, sigmaminor, beta, baseline, r2, modelpred, ...}`. The DN variant adds `srf_amplitude`, `srf_size`, `neural_baseline`, `surround_baseline`, `bold_baseline`. The HRF variant adds `hrf_delay`, `hrf_dispersion`. **All downstream comparison code keys off these exact file names** — don't rename them.

## Pipeline stages

```
prfsynth/install/      pull synthprf.sif (SLURM, singularityce)
prfsynth/synth/        generate synthetic BIDS dataset from prfsynth-{id}.json
prfsynth/prepare_data/ build BIDS from real data (vanes2019)
prfsynth/fit/{pkg}/    fit each PRF package on the BIDS dataset
prfsynth/analyze/      notebooks comparing fits against ground truth
```

The braincoder fitter is the only one driven by code in *this* repo:

- `prfsynth/fit/braincoder/docker_package/run.py` — the actual fitting script. Despite the folder name, the cluster SLURM scripts (`slurm_fit_braincoder_no_container_*.sh`) bypass Docker entirely and run this `.py` directly under `conda activate tf2-gpu`.
- `prfsynth/fit/braincoder/configs/*.yml` — fitting configs. Keys: `model.model_type`, `fitting.gauss_bounds` (mu_x, mu_y, sd ranges + `n_steps` for the initial grid), `fitting.n_gd_iterations`, `fitting.fit_hrf`, `fitting.divisive_normalisation`, `fitting.grid_only`, `fitting.l2_norm`, `fitting.force_positive_amplitude`, `synth.tr`, `synth.width`, `synth.height` (stimulus FOV in degrees — used to build the pixel→degree `grid_coordinates`).
- The other packages (aprf, popeye, mrvista, afni) run as upstream BIDS-app singularity/docker containers — the shell scripts in their folders only mount paths and pass configs.

### Three braincoder fit variants

| Variant | Config                          | SLURM script (A100)                                    | Output derivative dir                       |
|---------|---------------------------------|--------------------------------------------------------|---------------------------------------------|
| Gauss   | `default_config.yml` or `vanes2019_config.yml` | `slurm_fit_braincoder_no_container_A100.sh`            | `prfanalyze-braincoder.A100`                |
| Grid-only | `default_grid_config.yml` etc. | `slurm_fit_braincoder_no_container_A100_grid.sh`       | `prfanalyze-braincoder.grid.A100`           |
| HRF     | `fit_hrf_config.yml` / `vanes2019_hrf_config.yml` | `slurm_fit_braincoder_no_container_A100_hrf.sh`        | `prfanalyze-braincoder.hrf.A100`            |
| DN      | `vanes2019_dn_config.yml`       | `slurm_fit_braincoder_no_container_A100_dn.sh`         | `prfanalyze-braincoder.dn.A100`             |

There are parallel `_cpu.sh` and `_gpu.sh` (non-A100 GPU) variants of each SLURM script for the runtime benchmark; they produce `*.cpu` / `*.gpu` derivative dirs respectively. The notebooks in `prfsynth/fit/braincoder/logs/` (and the `runtime-*.txt` files) drive the speed-comparison figures.

## Running things

### Synthesize a dataset (cluster)

```bash
sbatch prfsynth/synth/generate_data_singularity.sh prfsynth/synth/prfsynth-small.json
# Drops sub-small under /shares/zne.uzh/gdehol/ds-prfsynth/BIDS
```

### Fit braincoder on the cluster (A100)

```bash
cd prfsynth/fit/braincoder
sbatch slurm_fit_braincoder_no_container_A100.sh vanes2019      # → prfanalyze-braincoder.A100
sbatch slurm_fit_braincoder_no_container_A100_dn.sh vanes2019   # DN only supported for vanes2019
sbatch slurm_fit_braincoder_no_container_A100_hrf.sh smallgrid
```

The conda env is `tf2-gpu` on the cluster (activated inside the SLURM script via `. "$HOME/.bashrc"; conda activate tf2-gpu`). This is the project's own env name, not one of the `*_cuda` envs listed in the global `~/.claude/CLAUDE.md`.

### Fit braincoder locally (Docker)

```bash
cd prfsynth/fit/braincoder
docker build -t prfanalyze-braincoder docker_package/
./fit_braincoder.sh small       # writes to /data/ds-prfsynth/BIDS/derivatives/prfanalyze-braincoder.cpu
./fit_braincoder_hrf.sh small   # writes to .../prfanalyze-braincoder.hrf
```

The Docker image is built from `docker_package/Dockerfile`. `fit_braincoder.sh` mounts `docker_package/run.py` into the container at `/run.py` and the chosen `configs/*.yml` at `/config.yml`.

### Prepare the vanes2019 dataset (real fMRI as fake BIDS)

```bash
python prfsynth/prepare_data/prepare_vanes.py -b /data/ds-prfsynth
# Requires: braincoder.utils.data.load_vanes2019 — i.e. the user's braincoder package installed.
```

### Run the comparison notebooks

```bash
cd prfsynth/analyze/notebooks
jupyter lab analyze_fits.ipynb   # compares all packages against ground truth
jupyter lab analyze_speed.ipynb  # runtime benchmark from logs/runtime-*.txt
```

These notebooks read `BIDS/derivatives/prfanalyze-{package}/...` directly and use `braincoder.utils.get_rsq` — they need the user's `braincoder` package on the path.

## Implementation gotchas

- **`run.py` works at a single-voxel-mask granularity, single session, single participant.** `assert(len(args.participant_label) == 1)` and `assert(len(sessions) == 1)` are enforced; pass exactly one identifier per invocation. The `--debug` flag masks ~1% of voxels for smoke tests.
- **Stimulus aperture orientation:** loaded as `(T, 1, Y, X)`, transposed to `(X, Y, T)` before being passed to `GaussianPRF2DWithHRF`. The Y axis is also **flipped** in `prepare_vanes.py` (`stimulus = np.transpose(...)[::-1, :, np.newaxis, :]`) so the BIDS aperture matches braincoder's coordinate convention. If you regenerate apertures by hand, replicate that flip or fits will land mirrored on Y.
- **Grid coordinates** are built from `synth.width`/`synth.height` (degrees of visual angle in the config YAML) and the *pixel* dimensions of the loaded aperture — so the FOV constants in the config must match what prfsynth used to render the apertures. Default synthetic FOV is 20×20 deg; vanes2019 is 20×11.
- **The fit pipeline in `run.py`** is: (1) grid search via `ParameterFitter.fit_grid` with correlation cost, (2) OLS refine of baseline + amplitude (`refine_baseline_and_amplitude`, with `l2_alpha=fitting.l2_norm`), (3) optional gradient descent (`par_fitter.fit(..., max_n_iterations=fitting.n_gd_iterations)`) **unless `fitting.grid_only: true`**. DN fitting reuses the Gauss OLS result as init, then fits the `DivisiveNormalizationGaussianPRF2DWithHRF` model. DN + HRF together raises `NotImplementedError`.
- **Output dtype caveat (see global ~/.claude/CLAUDE.md):** `save_as_nifti` here does *not* call `set_data_dtype(float32)` / `set_slope_inter(1, 0)`. Outputs may inherit a dtype/slope from the inferred mask — currently the mask is `np.ones(...)` (so float64-ish), which usually behaves, but if you ever swap in a real brain mask for non-synthetic data, add the float32 / slope-reset block before `to_filename`.
- **Parameter name mapping** in `run.py` (`pars` dict, lines ~49 and ~176) translates braincoder internal names (`x`, `y`, `sd`, `amplitude`, `baseline`) to BIDS-prfsynth derivative names (`centerx0`, `centery0`, `sigmamajor`, `beta`, `baseline`). The downstream notebooks expect the BIDS-side names — don't change one without the other.

## Side directories (not part of the prfsynth pipeline)

- `garcia2019/` — separate GLMsingle fit on `/data/ds-numrisk` (run `python garcia2019/fit_glm_single.py <sub> --bids_folder ...`). Has its own `Subject` class in `garcia2019/utils.py`; unrelated to prfsynth.
- `visualize/show_hcp99.py` — pycortex flatmap viewer for the vanes2019 derivatives, hard-coded to `subject = "hcp_999999"` and the `prfanalyze-braincoder.A100` derivative dir. Switch between Gauss/HRF/DN variants by editing the `model = ...` line. Requires a working `pycortex2` conda env (see global CLAUDE.md).
- `example_notebooks/simple_fit.ipynb` — minimal braincoder usage example.

## What to remember about external deps

- The user's own `braincoder` (`~/git/braincoder`) must be installed in whatever env runs `docker_package/run.py` or the analysis notebooks. The Dockerfile installs it from GitHub; cluster scripts assume it's in `tf2-gpu`. There is a dedicated `CLAUDE.md` inside `~/git/braincoder/` — read it for the encoding-model class hierarchy and optimizer pipeline.
- The competing packages (`prfanalyze-aprf`, `prfanalyze-popeye`, `prfanalyze-afni`, `prfanalyze-vista`) come as garikoitz/* docker images / cluster `.sif` files at `/shares/zne.uzh/containers/`. They aren't in this repo and aren't installed locally.

## Quick map into braincoder used here

The fitter in `docker_package/run.py` only touches a small slice of the braincoder API. If you need to extend it (new model, new fitter), here is where to look in `~/git/braincoder/braincoder/`:

| Symbol used here                                         | Defined in                          |
|----------------------------------------------------------|-------------------------------------|
| `GaussianPRF2DWithHRF`                                   | `models/prf_2d.py`                  |
| `DivisiveNormalizationGaussianPRF2DWithHRF`              | `models/prf_2d.py`                  |
| `SPMHRFModel`                                            | `hrf.py`                            |
| `ParameterFitter` (`fit_grid`, `refine_baseline_and_amplitude`, `fit`, `get_rsq`) | `optimize/` (was `optimize.py`)     |
| `load_vanes2019` (used by `prepare_data/prepare_vanes.py`) | `utils/data.py`                     |
| `get_rsq` (used in `analyze/notebooks/`)                 | `utils/` (re-exported)              |

The paper's wider framework (which the *paper* describes but this repo does not directly invoke) also includes `WeightFitter`, `ResidualFitter`, and `StimulusFitter` for Bayesian decoding — none of those run in this scaffolding; this repo is strictly the forward-encoding speed/accuracy benchmark.
